# Bug Assessment: Staleness evidence omits the baseline snapshot

- **Slug**: refresh-interval-stale-warning
- **Created**: 2026-06-15
- **Source**: pasted text and screenshots
- **Verdict**: valid
- **Severity**: medium

## Report (verbatim or summarized)

The original report claimed that a live playlist with a 4-second segment/target duration is
incorrectly reloaded after 2 seconds, causing a false staleness warning:

> Seems like continuity check is faulty. For some reason next refresh is sent not after 4s
> (segment length) but only after 2s. Which of course will cause stale-warning since new Live
> playlist has not yet been generated.

Relevant observations:

```text
[17:46:04.405] Fetch: 1080p_avc1-mp4a_83 HTTP 200; 14ms; 2.3 kB
[17:46:04.480] Refresh: 1080p_avc1-mp4a next in 2s
[17:46:06.583] Fetch: 1080p_avc1-mp4a_84 HTTP 200; 17ms; 2.3 kB
[17:46:06.587] Warning: 1080p_avc1-mp4a is stale.
[17:46:06.587] Warning 1080p_avc1-mp4a_84: Playlist has not changed for 6.4s
                  (target duration 4.0s)
[17:46:06.587] Evidence: playlists/1080p_avc1-mp4a/1080p_avc1-mp4a_83.m3u8
[17:46:06.587] Evidence: playlists/1080p_avc1-mp4a/1080p_avc1-mp4a_84.m3u8
```

Follow-up clarification:

> If second re-validation after 2s is correct, then the evidence must include all 3 playlists for
> this problem instance: initial, planned refresh, re-try refresh.

## Symptom

The 2-second retry and the stale warning are correct. However, the warning lists only snapshots
`_83` and `_84`. Those files prove that the planned refresh and retry were equal over approximately
2 seconds, but they do not prove the reported 6.4-second unchanged interval.

For this finding, the evidence must also include `_82`, the baseline snapshot from which the
no-change timer continued. The complete proof is therefore `_82` (last changed/baseline), `_83`
(planned refresh after one target duration), and `_84` (half-target retry where the threshold was
crossed).

## Reproduction

1. Monitor a live media playlist declaring `EXT-X-TARGETDURATION:4` with verbose events enabled.
2. Return a playlist body equal to the previous parsed `MediaPlaylist` at a scheduled reload.
3. Observe that the next delay is 2 seconds (`targetDuration / 2`).
4. Continue returning unchanged content until elapsed time since the last observed content change is
   greater than 6 seconds (`1.5 * targetDuration`).
5. Observe a `TOOL.staleness` warning.
6. Observe that evidence contains only `<id>_<n-1>` and `<id>_<n>`, omitting the last changed
   snapshot `<id>_<n-2>` that starts the measured stale interval in this scenario.

`_83` and `_84` are local zero-based refresh indices generated by `SnapshotID.label`; they do not
show that the origin media sequence or segment window advanced. `_82` is required to establish the
full timeline asserted by the finding.

## Root Cause Hypothesis

**Confidence: high.** Scheduling and staleness detection are correct. The evidence model incorrectly
assumes that every continuity finding is established by a single comparison of two consecutive
snapshots. Pairwise faults such as sequence regression fit that model; staleness does not. Staleness
asserts that content remained unchanged across a measured interval beginning at `lastChangedAt`.
For the shown warning, `_83` and `_84` cover only the retry interval. `_82` is the missing baseline
needed to substantiate the 6.4-second claim.

An Xcode `ExecuteSnippet` check with a 4-second target produced:

```text
initial: 4.0 seconds
changed: 4.0 seconds
unchanged: 2.0 seconds
```

## Proposed Remediation

**Preferred**: Keep `RefreshScheduler`, `StalenessDetector`, and the finding unchanged. Track the
refresh index associated with `lastChangedAt` alongside the timestamp. When a staleness threshold is
crossed at refresh `<n>`, resolve evidence for the closed snapshot range from the last changed
baseline through `<n>`. For the shown warning this is `_82`, `_83`, and `_84`.

Extend the internal evidence model with a sequence form that preserves order and can identify missing
captures. Keep `.pair` for genuinely pairwise continuity rules. Terminal and Markdown rendering must
show the same complete ordered sequence. The feature-003 evidence contract must be corrected because
its current "exactly two for every continuity finding" rule caused this limitation.

**Alternatives**:

- Special-case warning evidence as exactly `<n-2>`, `<n-1>`, `<n>`. Rejected as incomplete for error
  crossings or delayed fetches, where more snapshots may constitute the stale interval.
- Infer the baseline solely as `<n-2>`. Rejected because the number of retries before a threshold
  crossing is not a stable invariant.
- Change the cadence or threshold. Rejected because neither is the defect.

- Verify terminal and Markdown report evidence lists are identical and ordered.
- Preserve exactly two evidence files for pairwise continuity findings.

## Risks & Considerations

- The active feature declares evidence resolution and machine surfaces frozen. This user correction
  supersedes the pair-only evidence assumption, but the implementation should still avoid changing
  finding IDs, severities, messages, JSON schema, JSONL shape, or exit codes.
- The report currently recomputes evidence from `Finding` plus artifact paths. A complete fix must
  preserve the stale interval's baseline index for later report regeneration without broadening the
  frozen structured finding schema unintentionally.
- Long stale episodes can produce many evidence files. Rendering should remain complete, but the
  report and terminal need concise ordered labels so evidence remains readable.
- Missing captures inside a sequence must be explicit; silently collapsing to available files would
  again weaken the proof.
