# Valistream Constitution

## Core Principles

### I. Spec-First Development

No implementation work begins without an approved feature specification and
implementation plan in `specs/[###-feature-name]/`. Specifications MUST be
technology-agnostic, describe user value in prioritized user stories, and define
measurable success criteria. Every `[NEEDS CLARIFICATION]` marker MUST be resolved
before the plan phase completes; unresolved ambiguity MUST NOT be carried into
implementation.

**Rationale**: Code is the cheapest thing to change before it exists. Forcing scope,
priorities, and acceptance criteria into writing first prevents rework and makes
every later artifact reviewable against an agreed source of truth.

### II. Test-First (NON-NEGOTIABLE)

Every behavior change MUST be covered by tests written before the implementation:
write the test, confirm it fails, then implement until it passes (Red-Green-Refactor).
Test tasks are generated by default for every user story and MAY be omitted only when
the feature specification explicitly waives them with a recorded justification.
Tests MUST assert observable behavior, not implementation details.

**Rationale**: Tests written after the fact verify what the code does, not what it
should do. A failing test first is the only proof the test can fail at all.

### III. Simplicity and YAGNI

Start with the simplest structure that satisfies the current specification. New
projects, layers, abstractions, or patterns beyond direct need MUST be justified in
the plan's Complexity Tracking table, including why the simpler alternative is
insufficient. Speculative generality ("we might need it later") is not a valid
justification.

**Rationale**: Unjustified structure is permanent cost: every abstraction must be
learned, tested, and maintained. Complexity is accepted only when it pays for itself
against a present requirement.

### IV. Independent, Testable Increments

Work is sliced into user stories with explicit priorities (P1, P2, P3, …). Each story
MUST be independently implementable, independently testable, and deliverable on its
own; the P1 story alone MUST constitute a viable MVP. Cross-story dependencies that
break this independence MUST be eliminated or moved into the shared Foundational
phase.

**Rationale**: Independent slices keep the system shippable at every checkpoint,
allow parallel work, and make scope cuts a reordering decision rather than a rewrite.

### V. Observability and Versioning

Errors and key operations MUST emit structured, greppable log output; silent failure
is prohibited. Releases follow semantic versioning (MAJOR.MINOR.PATCH), and every
backward-incompatible change MUST ship a documented migration path before merge.

The version bump required by a backward-incompatible change depends on the release
series:

- **Stable series (>= 1.0.0)**: a backward-incompatible change MUST trigger a MAJOR
  bump.
- **Initial-development series (0.y.z)**: per SemVer, the 0.y.z series makes no
  public-API stability promise, so a backward-incompatible change MUST bump at least
  the MINOR (`0.y`) component — a MAJOR bump to `1.0.0` is NOT required — and MUST still
  ship the documented migration path. Releasing `1.0.0` is a deliberate stability
  declaration, never an automatic consequence of a breaking change.

**Rationale**: A system that cannot explain its own behavior cannot be debugged in
production, and consumers can only trust upgrades when version numbers carry a
contract. While a project is pre-1.0, the `0.y` line is itself the signal that the API
may still move; forcing a premature `1.0.0` to satisfy a literal MAJOR rule would
assert a stability the project has not chosen, so a migration-noted MINOR bump honors
the contract's intent without a false stability claim.

## Technology Constraints

The constitution is technology-agnostic; concrete stack decisions are made per
feature and recorded in the plan's Technical Context section (language/version,
dependencies, storage, testing tools, target platform, performance goals).

- A feature plan MUST NOT enter implementation with `NEEDS CLARIFICATION` remaining
  in its Technical Context.
- New runtime dependencies MUST be justified in the plan; prefer the standard library
  and already-adopted dependencies over new ones.
- Once a stack decision is recorded in a merged plan, later features MUST reuse it or
  amend this constitution / the prior plan with an explicit migration rationale.

## Development Workflow

Features flow through the Spec Kit pipeline in order:

1. `/speckit-specify` — create the feature specification (user stories, requirements,
   success criteria).
2. `/speckit-clarify` — resolve ambiguities; encode answers back into the spec.
3. `/speckit-plan` — produce the implementation plan and design artifacts. The
   Constitution Check gate MUST pass before Phase 0 research and be re-checked after
   Phase 1 design; violations MUST be recorded in Complexity Tracking or removed.
4. `/speckit-tasks` — generate dependency-ordered tasks grouped by user story.
5. `/speckit-analyze` — verify cross-artifact consistency before implementation.
6. `/speckit-implement` — execute tasks; commit after each task or logical group.

Code review MUST verify constitution compliance: reviewers reject changes that skip
the spec/plan gates, lack required tests, or introduce unjustified complexity.

## Governance

This constitution supersedes all other development practices in this repository.
Where guidance conflicts, the constitution wins.

- **Amendments**: Any change to this file is proposed as a pull request that updates
  the version line, prepends a fresh Sync Impact Report, and propagates required
  changes into the `.specify/templates/` files in the same change.
- **Versioning policy**: MAJOR for removing or redefining a principle in a backward
  incompatible way; MINOR for adding a principle or materially expanding guidance;
  PATCH for clarifications and wording fixes.
- **Compliance review**: Every implementation plan re-states the Constitution Check
  gates derived from this document and must pass them before and after design. Each
  pull request review confirms the gates were honored.
